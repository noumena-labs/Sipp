# 贡献指南

Sipp 是一个多语言 Monorepo 项目。提交贡献前，请确认目标明确、文档齐全，并用针对性最强的测试命令验证修改。

提交 Issue 或 PR 时，应能清楚说明修改的目的和工作原理。项目接受 AI 辅助编码（包括 Agent 生成的草稿），但作者须对最终代码变更负责——审查、理解并持续维护。

## 阐明原因

提交 Issue 或功能请求时，说明当前问题、受影响范围以及可能产生的影响。这有助于维护者评估优先级，规划合理的实现路径。

## 解释实现

提交 PR 时，描述修改了哪些内容以及具体实现方式。如果无法清晰解释代码行为、潜在风险和验证手段，请重新评估后再请求审查。

## 沟通交流

在 Issue 和 PR 中使用自己的语言。核心信息保持简明，仅在有助于审查者理解时补充细节。

每个 Issue 或 PR 应遵循以下原则：

- 解释修改的意义。
- 描述修改的具体内容。
- 保持变更原子化（单一职责）。
- 不做无关的代码清理。

## 动手开发前

- 阅读根目录 README 及相关的包或应用 README。
- 运行 `cargo xtask test list` 查看可用的测试目标。
- 用 `cargo xtask` 命令执行构建和耗时较长的工作流。
- 非任务明确要求，不要修改 `third_party/` 或 `crates/sys/llama.cpp/` 目录下的第三方代码。

## 文档规范

- README 保持简洁，以任务为导向。
- 详细的指南和参考资料放到本文档（mdBook）中。
- 优先提供可复制粘贴、能在全新克隆项目中直接运行的示例。
- 修改公共 API、包行为、命令或配置时，同步更新文档。

## 验证修改

仅修改文档时：

```bash
sipp docs build
cargo xtask test list
cargo xtask test verify --target public-docs
```

代码变更请运行[测试](testing.md)中针对性最强的测试目标。只有当修改跨越包或运行时边界时，才运行更全面的测试套件。
